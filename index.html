Memory leaks in Angular2

发表于 2017-12-17

问题描述：

初步验收时提的 bug: 看板页面太卡，用户在页面连续多次执行切换操作，会将卡顿传染到其他页面。调出 chrome 任务管理器，发现内存和cpu 占用率一直在涨。确定为系统存在内存泄露问题。

问题定位：

1、某个页面图表数据较多。整个渲染流程为:向后台请求数据 chartdata，然后 将 chartdata 赋值给图表组件（这一步就是渲染操作），由于后台返回的数据是很快的，那么意味着多个图表会同时渲染， 会导致cpu 占用率高的现象；

2、我们将图表限制为 6个，发现将指标放在 图表区域来回切换时（图表的 tooltip 会展示相应数据），还是会出现明显的卡顿现象；同时页面来回切换，内存和 cpu 占用率还是不停增长；

  • 使用 chrome 的开发者工具的 memory 快照，发现有些页面的内存在页面销毁时并没有释放；
  • 同时，detectchange操作耗时异常；

3、搜索引起 angular2 中的内存泄露问题的原因:

“你可能知道当你订阅 Observable 对象或设置事件监听时，在某个时间点，你需要执行取消订阅操作，进而释放操作系统的内存。否则，你的应用程序可能会出现内存泄露。”

项目中用到最频繁的是下拉框组件,尤其是支持多选时，用户一次选中的数据可能高达上百。当组件的 changeDetection 为默认时，组件的每次变化检测都会检查内部的所有数据（引用对象也会被深度遍历），这时非常耗时耗资源的，影响性能。而当值为 OnPush 时，组件的变化检测只检查输入属性的值（@Input 修饰的变量）是否发生变化。

export class MyThingComponent implements OnDestroy, OnInit {
    private ngUnsubscribe: Subject = new Subject();

    constructor(
        private myThingService: MyThingService,
    ) { }

    ngOnInit() {
        this.myThingService.getThings()
            .takeUntil(this.ngUnsubscribe)
            .subscribe(things => console.log(things));

        /* if using lettable operators in rxjs ^5.5.0
        this.myThingService.getThings()
            .pipe(takeUntil(this.ngUnsubscribe))
            .subscribe(things => console.log(things));
        */

        this.myThingService.getOtherThings()
            .takeUntil(this.ngUnsubscribe)
            .subscribe(things => console.log(things));

    }

    ngOnDestroy() {
        this.ngUnsubscribe.next();
        this.ngUnsubscribe.complete();
    }
}

引导语：数据处理不可避免，由于前期已定义的数据类型，或者数据展示的不同需求，经常需要对数据进行各种处理。本周的开发过程中，因为数据类型的不同，导致很多赋值操作的bug。下面总结一下 typescript/javascript 中数据类型转换的方式和处理数据注意点：

1.1 string 类型转换为 number 类型：

1.2 number 转换为 string 类型:

2 数据处理注意点

2.1 零值特殊处理

涉及到数据做除法时，需要注意除数是否为零值，做特殊处理。

2.2 值传递 引用传递触发 valueChange 事件

angular 中的是双向数据绑定，应该是引用改变触发 valuechange 函数。js 中的基本类型是值传递，而对象类型比如数组，应该使用会返回新数组的函数，而不是在原数组上改变，否则不能触发 valueChange 函数。
实例是开发过程中使用 highcharts 组件，给组件的 series 传值时，直接给原有对象赋值，但是图像初次没有重绘。

补充说明：js 中的参数传递类型

[1] JS 采取的都是”按值传递”的求值策略, 其中对象类型较为特殊，实际为按值传递了引用(即传递引用的副本，而不是按引用传递引用)。从这个角度，说对象也是按值传递也是有道理的。

[2] 引入“按共享传递”的求值策略，它让我们精确的区分按共享传递与经典的按值传递、按引用传递的不同。在这种情形下，可以按传参类型区分：“基本类型按值传递、而对象按共享传递。”（笔者更倾向的描述方式）

《用户体验要素》读后感

发表于 2017-09-24 | 分类于 读书笔记

一本小书，带你领略 Ajax 之父——Jesse James Gsrrett 眼中的用户体验。

《用户体验要素》基于“战略层，范围层，结构层，框架层和表现层”五个层面的基础架构来讲解用户体验。作者谈到整个软件产品设计过程中各个层面的一些注意事项，同时，为了避免混淆，还定义了一些业内便于交流的术语。另外，文中讲述了很多开发过程中一些看起来习以为常但实质是有理论依据的案例。整本书排版大方，内容简洁，结构分明。下面总结自己看书过程中的一点摘录和感悟。

产品设计 VS 用户体验设计

“外形服从于功能”，这种观点对于内部运作(用户不可见的部分)是适用的。但是，对于产品直接面对用户的那些部分，正确的产品形态绝不是由“功能”所决定的，而是由“用户自身的心理感受和行为”来决定的。

产品设计以及开发初级阶段，难免会有专注于功能而选择性忽略界面设计的嫌疑。这个时候在每一步的开发过程中就应该自己花心思来美化界面，通过一种帮助人们理解和接受的方式来呈现。感觉这个也需要一定平日积累，经常的情况是：“我知道这个地方很丑，但是不知道怎么改进。”

转化率

一个常用的投资收益的度量标准是转化率。通过跟踪弄有百分之多少的用户被你“转化”到了下一个步骤，就能衡量你的网站在达到“商业目的”方面的效率有多高。

刚好最近在做留存分析的界面，看到书中提到的转化率，不免心中有点点融会贯通的感觉。为了提高转化率，用户体验可以围绕“帮助人们工作得更快”和“减少他们犯错的几率”这两个方面进行改进。通过转化率可以量化用户体验对于用户留存的贡献值。

概念模型

用户对于“交互组件将怎样工作”的观点成为概念模型。

书中列举了两个例子：购物车和未改版前的西南航空公司网站。电子商务网站的购物车的图标能很形象地让用户明白它的作用，与之相反的是西南航空的网站：首页是一张客户服务的书桌的图片，书桌上一边堆着简介的小册子，另一边放着电话。因为过分运用概念模型，反而适得其反。脑补一下，你肯定觉得网站做成这样简直不可思议。
现实和概念，就像生活和艺术，概念源于生活，但要经过抽象后高于生活。

命名原则 nomenclature

好的命名原则可以辅助结构层让用户更好地找到他们想走的路。

  • 受控词典 controlled vocabulary

    受控词典是网站使用的一套标准语言。与用户谈话并了解他们的沟通方式，是开发出一个让用户感到自然的命名原则系统的最有效方式。

  • 类词词典 thesaurus

    类词词典会提供常用的、但未纳入该网站标准用语的词汇以供选择，你可添加内部专用术语、速写语、俚语或缩写词等与其相对应的词汇进行补充。

  • 元数据 metadata
    元数据是“关于信息的信息”，即以一种结构化的方式来描述内容的信息。构建受控词典和类词词典对于建立包含元数据的系统特别有用。

视觉词典 visual vocabulary

作者创建的图解网站结构的系统——视觉词典。详情可见——视觉词典文档。

用户习惯与布局

习惯使我们可以把反射作用应用到不同的环境中。

文中举例，有的制造商试着设计一些不符合“三行四列”的按钮布局的电话，结果每一次打电话时大约需要花3秒钟左右。一个不符合习惯的设计使得一件“下意识就能完成的事情”变成了“难以忍受的缓慢”。对于用户而言，这3秒钟不仅仅是时间的浪费，还有耐心的消磨和被挫折感占据的沮丧。在没有充分理由和明显益处时，不要轻易尝试强行通过界面改变用户习惯。

关于文档

当然，所有这些文档是大量的工作，但是有一个很好的理由值得去做：随着时间的推移，我们的决策原因会逐渐从记忆中消失。那些“在某种特殊环境下用来解决某个具体问题的临时决策”和那些“为了形成将来的设计工作的基础而有意识地做出来的决策”会混杂在一起。
